# 複数写真アップロード・複数日解析 MVP 仕様書

作成日: 2026-05-23  
フェーズ: Phase 2b（設計のみ、コード変更なし）  
対象アプリ: checker.html（栄養バランスチェッカー）

---

## 1. 機能の目的

| 項目 | 内容 |
|---|---|
| 主目的 | 複数日分・複数食分の写真をまとめて登録し、食事記録の作業効率を上げる |
| 副目的 | 1食ずつ登録する繰り返し操作を削減する |
| 対象ユーザー | 複数日の食事写真をまとめて記録したい利用者 |
| 用途の限界 | 参考記録のみ。診断・栄養判定・医療判断・疾病予防を目的としない |

**このMVPが解決すること**

現状の「写真」タブは1枚ずつしか解析できず、複数日分を検証する際に同じ操作を繰り返す必要がある。この機能では複数の写真を一度に選択・設定し、順番に解析してまとめて保存できるようにする。

---

## 2. MVP仕様

### 2-1. 基本仕様

| 項目 | 仕様 |
|---|---|
| 最大アップロード枚数 | 5枚（初期上限。将来的に変更可） |
| 対応形式 | `image/*`（既存 `photoFile` の accept 属性に準拠） |
| 解析方式 | 1枚ずつ逐次処理（並列送信なし） |
| 結果確認 | 全枚解析後に一覧で確認・修正してから保存 |
| 保存先 | `localStorage`（既存の `saveMeals()` を使用） |

### 2-2. 写真ごとのメタデータ設定

各写真に対してユーザーが以下を設定できる：

| フィールド | 入力形式 | デフォルト値 |
|---|---|---|
| 日付 | `<input type="date">` | 当日 |
| 食事区分 | `<select>`（朝食・昼食・夕食・間食） | 昼食 |
| メモ | `<input type="text">` または省略可 | 空 |

### 2-3. 解析ステータス

各写真は以下4つのいずれかの状態を持つ：

| status 値 | 表示ラベル | 意味 |
|---|---|---|
| `pending` | 未解析 | アップロード後、解析前 |
| `analyzing` | 解析中 | API呼び出し中 |
| `done` | 解析済み | 結果取得成功 |
| `error` | 失敗 | APIエラー / パースエラー |

### 2-4. ユーザー操作フロー概要

1. 「複数写真」タブ（または既存「写真」タブ内モード切替）を開く
2. 複数画像を一括選択（`multiple` 属性付き `<input type="file">`）
3. プレビュー一覧が表示される
4. 各写真に日付・食事区分・メモを設定する
5. 「まとめて解析」ボタンを押す
6. 1枚ずつ順番に解析が進む（進捗表示あり）
7. 完了後、全結果を一覧で確認・修正する
8. 「まとめて保存」ボタンで全件保存
9. `renderAll()` が呼ばれ、ダッシュボードに反映される

## 3. UI案

### 3-1. タブ配置の選択肢

**案A：既存「写真」タブ内にモード切替を追加**

```
[ フォーム ][ 写真 ][ テキスト ]
                ↓
  写真タブ内:
  [ 1枚モード | まとめてモード ← トグル ]
  
  1枚モード: 現状の写真入力UI
  まとめてモード: 複数写真UI
```

- 利点: タブ数を増やさない
- 欠点: 既存タブ内のHTML構造が複雑になる

**案B：新しい「まとめて写真」タブを追加**

```
[ フォーム ][ 写真 ][ テキスト ][ まとめて写真 ]
```

- 利点: 既存「写真」タブに変更なし。切り離して開発・テストできる
- 欠点: タブが1つ増える（モバイルで横並びが狭くなる）

**推奨: 案B**  
理由: 既存の1枚写真解析に影響を与えず、独立してMVPを実装・検証できるため。タブが増える問題は、モバイルではアイコン表示やスクロール対応で吸収できる。

## 4. 処理フロー詳細

### 4-1. アップロード〜解析開始

```
ユーザーが画像を選択
    ↓
files配列をBulkPhotoItemオブジェクトに変換
（id / file / previewUrl / date=今日 / mealType='lunch' / status='pending'）
    ↓
5枚超の場合は先頭5枚を採用し、超過枚数をトースト通知
    ↓
プレビューカード一覧を描画
    ↓
ユーザーが各写真の日付・食事区分・メモを設定
    ↓
「まとめて解析」ボタンを押す
    ↓
APIキーの有無を checkApiKey() で確認
APIキーなし → settingsModal を開いて中断
    ↓
解析ループ開始（下記 4-2）
```

### 4-2. 逐次解析ループ

```
items = status='pending' の写真一覧

for each item of items:
    item.status = 'analyzing'
    カードのUI更新（スピナー表示）
    
    try:
        base64 = await resizeImage(item.file)    ← gemini.js の既存関数
        result = await analyzePhoto(base64)       ← gemini.js の既存関数
        item.result = result
        item.status = 'done'
    catch err:
        item.error = err.message
        item.status = 'error'
        
        if err.message.startsWith('RATE_LIMITED') or
           err.message.startsWith('API_ERROR_503'):
            全体処理を中断（後続の写真は status='pending' のまま）
            中断トースト表示
            break
    
    カードのUI更新（完了 or エラー表示）
    
    if 次の写真がある:
        await sleep(1500)    ← 1.5秒待機（無料枠対策）

全ループ完了後:
    解析結果確認エリアを表示
    「まとめて保存」ボタンを有効化
```

### 4-3. 保存

```
ユーザーが栄養素値を確認・必要に応じて編集
    ↓
「まとめて保存」ボタンを押す
    ↓
status='done' の写真のみ対象
    ↓
各itemのresult + date + mealType + memoから
既存Mealオブジェクト形式に変換
    ↓
saveMeals( loadMeals().concat(newMeals) )
    ↓
renderAll()
    ↓
「N件保存しました」トースト
    ↓
アップロードエリアをリセット（オプション）
```

## 5. API無料枠対策

| 対策 | 実装方針 |
|---|---|
| 枚数上限 | MVP初期は最大5枚。`MAX_BULK_PHOTOS = 5` で定数管理 |
| 逐次処理 | 並列送信なし。for ループで1枚ずつ `await` |
| 待機時間 | リクエスト間に `await sleep(1500)` を挿入（調整可能な定数） |
| 429停止 | `RATE_LIMITED:` エラーで後続処理を中断 |
| 503停止 | `API_ERROR_503` エラーで後続処理を中断 |
| 再試行 | 失敗した写真のみ個別に再試行できる。全体再解析は行わない |
| 事前警告 | 解析開始前に「無料枠では1分あたりの解析件数に上限があります」と表示 |
| モデル選択 | 既存の `eiyou_model` localStorage 値（`gemini-2.5-flash` 等）をそのまま使用 |

## 6. エラー時の挙動

| エラー種別 | 発生条件 | 挙動 |
|---|---|---|
| APIキー未設定 | `getApiKey()` が空 | `settingsModal` を開いて処理中断 |
| APIキー無効 | `API_KEY_INVALID` | トースト表示、その写真を `status='error'` に |
| 429 レート制限 | `RATE_LIMITED:` | トースト表示、後続写真を含め全体処理を中断 |
| 503 高需要 | `API_ERROR_503` | トースト表示、後続写真を含め全体処理を中断 |
| JSONパース失敗 | `result` のパースエラー | その写真のみ `status='error'`、後続を継続 |
| 画像読み込み失敗 | `resizeImage()` の例外 | その写真のみ `status='error'`、後続を継続 |
| 途中キャンセル | 「中断」ボタン押下 | `cancelGemini()` 呼び出し、現在処理中の写真を `status='error'` に |
| 一部のみ成功 | done/error が混在 | done のみ保存対象。保存ボタン押下時に「N枚中M枚が保存対象です」と明示 |

## 7. 既存機能との関係

### 7-1. 変更しないもの

| 既存機能 | 方針 |
|---|---|
| 「写真」タブの1枚解析 | **変更しない**。案Bでは完全に独立したタブ |
| `analyzePhoto()` | **そのまま再利用**。引数は `imageBase64` のみ（変更不要） |
| `resizeImage()` | **そのまま再利用**。引数は `(file, maxBytes)` |
| `handleApiError()` | **直接呼ばない**。エラー文字列を解析してBulkPhotoItem単位で処理 |
| `saveMeals()` | **そのまま再利用**。既存Mealオブジェクト形式で保存 |
| `renderAll()` | **そのまま呼び出す** |
| `checkApiKey()` | **そのまま再利用** |
| `cancelGemini()` | **そのまま再利用**（中断ボタン用） |
| `MEAL_TYPES` | **そのまま参照** |
| APIキー保存方式 | **変更しない** |

### 7-2. 既存との差異

| 差異点 | 既存（1枚） | 今回（複数） |
|---|---|---|
| 確認モーダル | `showConfirmModal()` でモーダル表示 | インライン確認エリア（モーダル多重起動を避ける） |
| 保存タイミング | 確認後すぐ保存 | 全枚確認後に「まとめて保存」 |
| エラー処理 | `handleApiError()` でトーストのみ | ステータス別UI + 一部エラーで全体中断 |

### 7-3. 既存 bulk.js との関係

`checker/bulk.js` はテキスト一括解析（`analyzeBulkText()`）の処理を担当しており、今回の写真一括解析とは別機能。  
新機能は `checker/bulk-photo.js`（新設）として分離し、`bulk.js` を変更しない。

## 8. データ構造案

### BulkPhotoItem オブジェクト（一時管理用、localStorage保存対象外）

```javascript
{
  id:         string,     // 一意ID（例: 'bulk-photo-' + Date.now() + '-' + index）
  file:       File,       // 元のFileオブジェクト（保存しない）
  previewUrl: string,     // URL.createObjectURL(file) の結果
  date:       string,     // 'YYYY-MM-DD' 形式
  mealType:   string,     // 'breakfast' | 'lunch' | 'dinner' | 'snack'
  memo:       string,     // ユーザー入力メモ（空文字も可）
  status:     string,     // 'pending' | 'analyzing' | 'done' | 'error'
  result:     object|null,// analyzePhoto() の戻り値（nutrients 等）
  error:      string|null,// エラー時のメッセージ
  createdAt:  number      // Date.now()
}
```

### 保存時に変換するMealオブジェクト（既存形式）

```javascript
{
  id:          string,   // 既存形式のID
  date:        string,   // BulkPhotoItem.date
  mealType:    string,   // BulkPhotoItem.mealType
  inputMethod: 'photo',  // 既存の inputMethod 値に合わせる
  memo:        string,   // BulkPhotoItem.memo
  foods:       array,    // result.foods
  nutrients:   object,   // result.nutrients（ユーザーが編集した値）
  createdAt:   number    // Date.now()
}
```

## 9. 採用しない案

| 案 | 採用しない理由 |
|---|---|
| 複数写真を1回のAPIリクエストにまとめる | 無料枠のレート制限・トークン上限に達しやすい。JSONパース失敗時の再試行コストが高い |
| 無制限アップロード | 無料枠では非現実的。UX上も枚数が多いほど待機時間が長くなる |
| 自動で日付・食事区分を確定する | 写真のメタデータ（Exif等）は取得できる場合でも、ユーザーが意図した日時と異なる可能性がある |
| 解析結果を確認なしで自動保存 | 誤解析・AI推定誤りをユーザーが確認できなくなる |
| サプリメント販売導線の追加 | アプリの目的と合致しない |
| 背景自動解析（ページ読み込み時） | ユーザーの意図しないAPI呼び出しが発生する |
